# Moqi 3.0 架构

## 设计目标

Moqi 解决的不是“把用户全部写给 AI 看”，而是让 AI 以最低的上下文成本获得足够信息，做出更符合用户目标、判断和边界的行动。

## 三层架构

### 入口层：entrypoints

承载每轮都必须执行的规则：沟通格式、行动边界、安全红线、通用执行原则和任务路由。

- `AGENTS.md` 是跨 agent 共享规则的唯一真实源。
- `CLAUDE.md` 等平台文件只负责转入共享入口和记录平台差异。
- 项目级入口可以细化规则，但不能降低全局安全边界。

### 核心层：MSA

- **Memory**：当前且会改变协作行动的事实、优先级和资料真实源。
- **Soul**：有证据支撑、更新缓慢的思维模式和判断框架。
- **Agent**：协作关系、纠错态度、主动程度和分歧处理。

MSA 不是个人档案。教育、履历、文章数量和完整项目清单应放在外部真实源，Memory 只保存指针。

### 细则层：references

承载只在特定任务中需要的规则，例如 Git、代码、调试、研究证据和 Roadmap。每个 reference 必须在主入口有明确触发条件。

## 两个支撑平面

### ALIGNMENT 控制平面

负责信息分流、去重、过期清理、Gotchas 生命周期、结构验证和范围明确的提交。它不属于 MSA，也不是普通领域 reference。

### Skills 能力平面

承载完整、可复用、有稳定触发边界的工作流。实例只维护触发和源码管理规则，不复制 Skill 全文。

## 单一真实源

同一事实或规则只在一个位置完整表达：

- 入口可写安全摘要，reference 展开操作步骤。
- README 可以解释架构，但不定义新的运行规则。
- Git 保存历史，常驻文件只保留当前有效状态。
- Gotcha 是严重事故的临时高优先级提醒，达到退出条件后删除。
