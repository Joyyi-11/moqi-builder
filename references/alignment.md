# Align 模式

## 目标与触发

对齐不是追加一次会话记录，而是把跨会话仍有价值的变化同步到唯一真实源，同时删除重复、过期和失效内容。

先读取实例中的 `ALIGNMENT.md`。实例规则可以细化本流程，但不能降低文件保护、提交和发布边界。只有用户明确触发对齐时执行；普通讨论和只读诊断不自动写文件。

## 一、建立基线

1. 从磁盘重读入口、`Memory.md`、`Soul.md`、`Agent.md`、所有 references、`ALIGNMENT.md` 和相关项目文档。
2. 记录 Git 状态、已有未提交内容、入口链接和资料真实源。
3. 区分对齐前已有修改、本轮产生的修改、生成物和来源不明文件。
4. 明确本次是否仅更新内容，还是包含结构调整、链接切换或本地提交。

对齐不能自动接管、覆盖或暂存对齐前已有的未知修改。

## 二、提取有效变化

通读当前会话和已确认材料，只提取跨会话仍会改变行动的信息。以下内容不写入：

- 一次性命令、操作流水和临时中间状态。
- 未经用户确认的推测。
- 已有权威资料源可以按需读取的完整资料。
- 只是措辞变化、没有行为差异的规则。
- 已经被另一条规则完整覆盖的重复内容。

## 三、信息分流

| 信息类型 | 唯一写入位置 |
|:---------|:-------------|
| 每轮必须执行的行为、红线和任务路由 | `entrypoints/AGENTS.md` |
| 平台独有的加载方式和差异 | 对应平台适配器 |
| 会改变当前协作判断的事实和资料源 | `MSA/Memory.md` |
| 多个真实场景支撑的稳定思维模式 | `MSA/Soul.md` |
| 协作关系、主动程度、纠错和分歧处理 | `MSA/Agent.md` |
| 有明确任务触发条件的低频细则 | 对应 `references/*.md` |
| 系统维护、信息分流和清理机制 | `ALIGNMENT.md` |
| 一次性过程、未验证信息和操作流水 | 不写入 |

完整可复用的专业流程不复制进 Moqi；只记录它的触发边界或权威来源。

## 四、克制更新和去重

1. 优先改写现有条目，其次合并或删除，最后才新增。
2. 检查同一规则是否同时存在于入口、MSA、references、README 或平台适配器。
3. 允许入口写原则、reference 展开操作步骤；不允许逐句复制。
4. 删除相对时间、失效路径、过期项目状态和无法再影响判断的事实。
5. `Soul.md` 更新门槛最高；单次情绪、临时偏好和单一事件不形成稳定模式。
6. 更新入口或文件位置后，同步检查所有路由、适配器、链接和校验器。

## 五、Gotchas 生命周期

Gotcha 必须包含：发生了什么、已确认的原因、以后必须怎样做。

- 只记录真实发生、影响严重或重复出现，而且尚未被结构机制解决的问题。
- 活跃 Gotchas 最多 3 条。
- 根因未知时先调查，不把猜测写成原因。
- 稳定规则已经吸收纠正动作时，Gotcha 只能作为有退出条件的临时提醒存在。
- 根因被结构性消除，或达到实例约定的连续正确执行次数后删除。
- 历史由 Git 保存，不建立永久踩坑档案。

## 六、结构验证

运行实例校验器，并人工确认：

- `AGENTS.md` 仍是共享运行规则的唯一真实源。
- `Memory.md`、`Soul.md`、`Agent.md` 没有越界或互相重复。
- 每个 reference 都有明确且唯一的入口路由。
- 平台适配器能加载主入口，不复制共享规则。
- 没有未替换占位符、失效路径和孤立文件。
- 本轮 diff 没有覆盖对齐前的未知修改。

## 七、提交边界

只有用户明确授权提交，或实例把精确触发词定义为包含本地提交时，才创建 commit。

1. 列出本轮实际修改文件。
2. 使用 `git add -- <file...>` 精确暂存，不使用 `git add .`。
3. 检查 `git diff --cached --check` 和完整 staged diff。
4. 创建范围明确的本地 commit。
5. 提交后报告剩余工作区状态。

对齐不自动 push、打 tag、创建 release、部署或公开发布。

## 八、交付

报告已更新、已删除、未处理和待确认内容；给出验证结果、提交状态、入口状态和仍需用户决定的旧路径处置。
