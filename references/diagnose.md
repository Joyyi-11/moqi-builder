# Diagnose 模式

## 审计顺序

1. 入口和链接是否真实可用。
2. 文件职责是否越界。
3. 内容是否重复、冲突或过期。
4. 条件路由能否触发所有 references。
5. MSA 是否只保留协作必需信息。
6. ALIGNMENT 是否形成更新、验证和清理闭环。

## 文件检查

### AGENTS

- 是否能独立约束关键行为和红线？
- 是否塞入平台版本、完整 prompt、历史叙事或个人履历？
- 是否为每个 reference 提供明确触发条件？

### Memory

- 每条事实是否仍会改变当前协作？
- 动态数字、完整经历和工具清单是否可以改为外部资料指针？
- 是否存在“正在”“最近”等会迅速失效的状态？

### Soul

- 每个模式是否有反复证据？
- 是否混入运行规则、系统设计、临时观察或待办？

### Agent

- 是否只描述协作关系、纠错和分歧处理？
- 是否复制入口规则、文件索引、Skill 快捷词表或维护流程？

### References / Alignment / Skills

- references 是否按任务加载并保持短小？
- alignment 是否位于控制平面而不是 MSA 内部？
- 已安装 Skill 是否还在实例中保留第二份 prompt？
- Gotchas 是否仍有原因、动作和退出条件？

## 输出

 findings 按风险排序，给出文件、证据、归属和具体处理方式。区分“必须修复”“建议精简”“需要用户确认”。诊断模式不自动修改文件。
