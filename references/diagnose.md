# Diagnose 模式

## 审计顺序

1. 入口和链接是否真实可用。
2. 文件职责是否越界。
3. 内容是否重复、冲突或过期。
4. 条件路由能否触发全部现有 references。
5. MSA 是否只保留协作必需信息。
6. `ALIGNMENT.md` 是否形成更新、验证和清理闭环。

## 文件检查

### `AGENTS.md`

- 是否能独立约束关键行为和红线？
- 是否包含日常识别跨会话变化、主动提议、用户确认后更新真实源的共同维护闭环？
- 是否混入平台版本、完整工作流、历史叙事或个人履历？
- 是否为每个现有 reference 提供明确且唯一的触发条件？

### `Memory.md`

- 每条事实是否仍会改变当前协作？
- 动态数字、完整经历和工具清单是否应改为外部资料指针？
- 是否存在“正在”“最近”等已经失效的相对状态？

### `Soul.md`

- 每个模式是否有多个真实场景支撑？
- 是否混入运行规则、系统设计、临时观察或待办？

### `Agent.md`

- 是否只描述协作关系、主动程度、纠错和分歧处理？
- 是否复制入口规则、文件索引、工具清单或维护流程？

### references 和 `ALIGNMENT.md`

- 每个 reference 是否确有重复任务场景，且只按需加载？
- 没有 reference 的用户是否被强行生成了通用技术细则？
- `ALIGNMENT.md` 是否包含基线、分流、去重、Gotchas、验证和提交边界？
- Gotchas 是否仍有真实原因、纠正动作和退出条件？

## 输出

按风险排序给出文件、证据、正确归属和具体处理方式。区分“必须修复”“建议精简”“需要用户确认”。诊断模式不自动修改文件。
