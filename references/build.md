# Build 模式

## 目标

通过分阶段对话建立一个可运行的 Moqi 实例。采集会改变 AI 判断与行动的信息，不追求完整记录用户。

## Phase 0：确认起点和边界

先检查用户是从零开始，还是已有入口、个人说明、规则文件或协作记录。

用 AMV 向用户复述并确认：

- **A（当前）**：主要 agent、操作系统、目标目录、已有文件、入口和工作区状态。
- **V（目标）**：最终需要哪些文件，怎样算搭建成功，哪些信息暂不处理。
- **M（路径）**：分轮访谈、起草确认、写入、链接切换和验证顺序。

同时确认行动授权、隐私边界，以及是否允许修改全局入口。说明预计文件清单后再开始采集。

如果已有系统，先审计并分类现有内容，不直接套模板覆盖：

- 当前仍有效，可以原位改写。
- 有价值但归属错误，需要迁移。
- 重复、过期或只有历史价值，建议删除。
- 来源不明或判断不足，保留并标记待确认。

## Phase 1：当前协作事实

目标是形成 `Memory.md`。从这个问题开始：

> AI 在接下来一段时间必须知道什么，才不会做出错误判断或让你重复解释？

顺着用户回答追问当前优先级、活跃上下文、稳定资源和权威资料源。只有信息会改变 AI 行动，而且没有更合适的外部真实源时，才写入正文。

不要默认索取完整学历、履历、技能清单、工具列表和动态数字。涉及这些内容时，优先记录最新资料的路径或 URL。

结束条件：已经能回答“AI 当前必须知道什么”，且用户确认没有关键遗漏。

## Phase 2：稳定思维模式

目标是形成 `Soul.md`。不要直接问“你有什么思维模式”，使用真实场景：

- 最近一次面对陌生问题时，你先做了什么？
- 做重要判断时，什么证据会让你改变看法？
- 学习、写作或做项目卡住时，你通常怎样推进？
- 哪些选择看似矛盾，却长期同时存在？

每个候选模式至少需要两个不同真实场景支撑。区分：

- 用户明确表达的自我判断。
- 从多个事实归纳出的模式。
- 只有单一证据、仍需确认的推测。

证据不足的内容不写成稳定结论。结束条件：已有少量高解释力模式，用户认可其表述和证据。

## Phase 3：协作关系

目标是形成 `Agent.md`。围绕具体协作体验询问：

- 希望 AI 是怎样的同事或伙伴？
- AI 应该主动到什么程度，什么时候必须停下来问？
- AI 发现偏好变化、重复纠正或规则失效时，是否应主动建议更新协作系统？什么信号才足以提出？
- AI 犯错、被纠正或与用户有分歧时应该怎样处理？
- 哪些互动最令人反感？
- 需要共同判断时，希望 AI 追问、质疑还是直接给建议？

把关系定位、主动程度、纠错方式和分歧处理写入 `Agent.md`。每轮必须执行的格式要求、授权边界和安全红线转入 `AGENTS.md`。

结束条件：用户能明确判断“照这样协作会不会舒服、可信且有效”。

## Phase 4：运行规则和按需细则

从用户真实任务中识别高频摩擦和高风险动作：文件操作、Git、发布、外部平台、研究证据、写作或其他领域。

逐条判断归属：

- 不执行就会立即出错或产生高频摩擦：`AGENTS.md`。
- 日常协作中识别跨会话变化、提出修改建议并在确认后更新真实源的维护闭环：`AGENTS.md`。
- 只在某类任务中需要，而且内容足够形成独立细则：创建对应 `references/*.md` 并添加入口路由。
- 已有专门工具或 Skill 能完整完成：只配置触发边界，不复制其流程。
- 只是一次性偏好或本轮步骤：不写入系统。

没有明确需求时不创建 references，不用技术用户的模板替代访谈结果。

## Phase 5：逐份起草和确认

按以下顺序交付草稿，一次确认一份：

1. `Memory.md`
2. `Soul.md`
3. `Agent.md`
4. `AGENTS.md`
5. `ALIGNMENT.md` 和确有必要的 references

每份草稿同时列出：已确认内容、模型归纳、证据不足和待用户决定的部分。用户纠正时修改原条目，不通过追加新规则掩盖冲突。

全部内容确认后，再列出实际写入文件、平台适配器、链接变化和验证动作，获得执行授权。

## Phase 6：生成、切换和验证

1. 从 `assets/templates/` 选择核心骨架，不整目录盲目复制。
2. 依据确认内容替换或删除占位符，不保留无意义空章节。
3. 只为用户实际使用的平台生成适配器；Claude Code 可选用 `assets/adapters/CLAUDE.md`。
4. 将 `scripts/verify_system.py` 复制到实例的 `scripts/verify_system.py`。
5. 如果存在 references，为每个文件添加唯一、清晰的 `AGENTS.md` 路由。
6. 写入后逐份回读，确认没有推测被写成事实。
7. 运行 `python scripts/verify_system.py <moqi-root>`。
8. 先验证新入口能加载，再请求用户确认；旧源删除必须另行授权。

交付时报告文件清单、入口关系、验证结果、待确认项和首次 `[对齐]` 的使用方式。
