# Build 模式

## 目标

从零建立一个可运行的 Moqi 实例。采集的是“会改变协作行为的信息”，不是完整个人档案。

## 一、确认环境

1. 确认主要 agent、操作系统、期望存放路径和是否需要跨平台入口。
2. 检查目标目录、现有全局入口、项目规则和 Skills 路径。
3. 说明将创建的文件清单、链接关系和验证方式，获得确认。

## 二、采集四类信息

每轮只聚焦一个主题，顺着回答追问，不照表审问。

### 当前协作事实

问：AI 在接下来一段时间必须知道什么，才不会做出错误判断？

只采集当前优先级、活跃项目、稳定资源和权威资料路径。学历、履历、工具清单和动态数字默认不进 Memory。

### 思维模式

用真实场景询问用户如何学习、判断、写作、处理不确定性和做取舍。Soul 中每个模式至少有两个不同场景的证据，证据不足则放入待确认，不写成结论。

### 协作关系

询问 AI 的角色、主动程度、纠错方式、分歧处理和用户最反感的互动。关系偏好进入 Agent；每轮都必须执行的行为和红线进入入口。

### 运行规则

询问行动授权、文件删除、Git、生产平台、隐私和公开发布边界。再结合用户常见任务确定需要哪些 references 和 Skills。

## 三、生成实例

1. 复制 `assets/templates/` 到用户确认的 Moqi 根目录。
2. 将 Builder 的 `scripts/verify_system.py` 复制到实例的 `scripts/verify_system.py`。
3. 替换 `{{USER_NAME}}`、`{{MOQI_ROOT}}`、`{{PRIMARY_AGENT}}` 等占位符。
4. 按采集结果改写 MSA，不保留无意义的空章节。
5. 只生成确有触发场景的 references；同步更新 AGENTS 路由。
6. 建立平台入口链接前说明影响，获得确认后执行。

## 四、验证与交付

运行：

```bash
python scripts/verify_system.py <moqi-root>
```

交付时报告文件清单、入口关系、未确认信息、验证结果和首次 `[对齐]` 的使用方式。
