# `[对齐]` 工作流

精确触发 `[对齐]` 后，对入口、MSA、references 和相关项目文档做全局检查。对齐的目标是同步有效变化并持续减法，不是追加会话日志。

## 一、建立基线

1. 从磁盘重读全部受影响文件和当前会话。
2. 记录 Git 状态、已有修改、入口链接和资料真实源。
3. 区分对齐前已有内容、本轮修改、生成物和未知文件。
4. 未经确认不覆盖、回滚或暂存已有未知修改。

## 二、信息分流

| 信息类型 | 唯一写入位置 |
|:---------|:-------------|
| 每轮必须执行的行为、红线和路由 | `entrypoints/AGENTS.md` |
| 当前协作事实和资料源 | `MSA/Memory.md` |
| 多个场景支撑的稳定思维模式 | `MSA/Soul.md` |
| 协作关系、纠错和分歧处理 | `MSA/Agent.md` |
| 有明确任务触发条件的低频细则 | 对应 `references/*.md` |
| 系统维护和信息分流机制 | `ALIGNMENT.md` |
| 一次性过程、流水和未验证信息 | 不写入 |

## 三、克制更新

- 优先改写、合并和删除，最后才新增。
- 同一规则只保留一个真实源；入口原则和领域步骤不逐句复制。
- 删除过期状态、失效路径、相对时间和不再影响判断的事实。
- `Soul.md` 不根据单一事件形成稳定结论。
- reference 只有在确有重复任务场景且入口存在路由时才保留。

## 四、Gotchas

Gotcha 必须写清发生了什么、已确认的原因、以后必须怎样做。

- 只记录真实发生、影响严重或重复出现且尚未解决的问题。
- 活跃 Gotchas 最多 3 条，根因未知时不猜。
- 稳定规则吸收纠正动作后，只能作为有退出条件的临时提醒。
- 根因被结构性消除或达到连续正确执行条件后删除，历史由 Git 保存。

## 五、验证和提交

运行实例校验器，检查文件边界、重复、路由、适配器、链接、占位符和本轮 diff。

只有获得授权时才精确暂存本轮文件、检查 staged diff 并创建本地 commit。不自动 push、tag、release、部署或公开发布。

## 六、交付

报告已更新、已删除、未处理和待确认内容，以及验证、提交、入口和旧路径状态。
